Project 1: Threads

doc

In this project, a very basic thread system is provided, and we need to extend its functionality.

Alarm Clock

  • Problem: The existing timer_sleep() function (in devices/timer.c) uses busy waiting, continuously looping until sleep ends.

To solve this problem, we can consider adding threads to a waiting_list queue (sorted by wakeup order) when they enter sleep, and thread_block() them. The interrupt checks at each clock cycle whether the list head should wake up, and thread_unblock() them one by one until the list head no longer needs to wake up.

The approach is simple; the rest is about code implementation details:

  1. We need to be aware of concurrency conflicts between normal threads and interrupt threads. Since waiting_list needs to be used in both timer_sleep() (called by normal threads) and timer_interrupt() (called by interrupt threads), we need to disable interrupts before operating on this queue in timer_sleep().
  2. You'll find that one alarm-priority test case consistently fails. Reading the test case C code reveals it requires priority scheduling (part 2 of this project - I still don't understand why this part is placed before Priority Scheduling). Its impact on this part's code is that whenever a thread is awakened in the interrupt thread, you need to call intr_yield_on_return() (i.e., put the currently running thread back into ready_list and retrieve according to priority).

Additionally, you'll find that without making any modifications to the provided PintOS, you can still pass all test cases in part 1...

Priority Scheduling

  1. Standard: Implement a strict priority scheduling mechanism.
    • When a higher priority thread is added to the ready_list, the currently running lower priority thread needs to thread_yield() to give up CPU resources.
    • When multiple threads are waiting for a lock, semaphore, or condition variable, once the resource is released, the waiting thread with the highest priority should be awakened first.

This part is more complex than the previous one, mainly because it introduces many ambiguous terms. We need to understand what locks, semaphores, and condition variables are.

Overall, we can order these three terms from low-level to high-level: semaphore -> lock -> condition variable.

Let's briefly introduce these three concepts.

semaphore

The core is a counter + blocking queue.

struct semaphore {
    unsigned value;
    struct list waiters;
}

It can manage a limited number of resources.

We can use sema_down(s) and sema_up(s) functions to acquire and release resources. If no corresponding resource is available during acquisition, it blocks until a resource is provided to that thread.

lock

Essentially a semaphore(1).

For example, if we allocate a piece of shared memory that multiple threads can operate on, we must lock this memory to avoid race conditions.

condition variable

Multiple threads wait to be awakened after a certain condition is met.

Now, with a basic understanding of these three terms, consider this situation: if a lock is owned by a low-priority thread, and a high-priority thread enters the ready_list and wants to own this lock, due to semaphore constraints, it needs to wait for the low-priority thread to finish running and release the lock before it can be awakened. If a bunch of medium-priority threads that don't need this lock join at this time, the high-priority thread that should have run first will never run - this is not what we want to see.

This leads to the second task of this part:

  1. Priority Donation
    • After a thread owns a lock, its priority should be the priority of the held lock (define the lock's priority as the maximum priority among threads in the lock's wait queue and the holding thread).
    • Additionally, we can easily see that recursive situations will occur, so the priority of all threads in the entire dependency chain should be adjusted.
    • When a lock is released, the priority of the thread that originally held the lock should be updated, and it may be thread_yield()ed.

The code implementation is not particularly difficult; the challenge lies in understanding the above concepts (I'm not sure if my explanation is clear enough).

Advanced Scheduler

  • Implement a Multilevel Feedback Queue Scheduler (MLFQS) similar to the 4.4 BSD operating system to reduce the system's average response time.

What is BSD?

The goal of the BSD scheduler is to automatically achieve fair scheduling and good interactive responsiveness without external priority information.

The key points are:

  1. Dynamic Priority

    Unlike simple priority schedulers that require manual priority setting, the BSD scheduler automatically adjusts priority based on thread behavior:

    • CPU-intensive tasks (heavy computation) → priority decreases
    • I/O-intensive tasks (heavy interaction) → priority increases

  2. Metrics

    • recent_cpu (recent CPU usage)
    • nice value: user-settable (-20~20), higher nice value = more "friendly" = lower priority
    • load_avg (system load, affects recent_cpu decay rate)
